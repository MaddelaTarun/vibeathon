### Missing Questions

1. **How does the Customer enter their table code?** — The RoleSelect shows a table number input, but there's no way for a customer to "join" an existing table created by a waiter. Should the waiter verbally give the 4-digit code, or should it be on a printed slip? *Matters because: it determines whether the table code is auto-generated by waiter or customer.*

2. **Can a customer see the menu and choose their chef, or only the waiter?** — Currently the WaiterView has menu browsing + chef selection. The CustomerView only tracks orders. *Matters because: it changes the Customer UI scope significantly.*

3. **What happens when all chefs in a cuisine category are at max load?** — `autoSelectChef()` in `server/chefs.js:181-186` returns `null` when all are full. The order is created with `chefId: null`. *Matters because: the customer gets an order with no assigned chef and no ETA.*

4. **Should reviews be editable after submission?** — Currently once submitted, the review is permanent (`reviews.js` has no update method). *Matters because: accidental submissions can't be corrected.*

5. **Is there a "chef" role login?** — `SessionContext.tsx:3` defines `'chef'` as a valid role, and `sessions.js:20` supports it, but there is no `ChefView.tsx` page. *Matters because: chefs have no way to see their assigned orders or update order status.*

---

### Undefined Guardrails

1. **Rating bounds** — `reviews.js:19-22` clamps to 1-5, which is correct. But the frontend star selector (`CustomerView.tsx:216`) starts at 1 but initializes at 5 (`reviewData` default is all 5s at line 36). This biases ratings upward. **Suggested fix:** Default to 0 (unrated) and require all dimensions before submitting.

2. **Table code collision** — `sessions.js:43` generates a 4-digit random code. With 9000 possibilities, collisions are likely at scale. **Suggested bound:** Validate uniqueness before assigning; or use 6-digit alphanumeric.

3. **Max cart size** — Nothing prevents a waiter from adding 100 items. **Suggested bound:** Limit cart to 20 items.

---

### Scope Risks

1. **Chef View creep** — The user mentioned "chefs will be selected based on reviews" which implies chef-side features. Building a full ChefView is a separate phase — keep it out of this sprint. **Prevention:** Explicitly exclude ChefView from Phase 1.

2. **Customer ordering directly** — The user said "orders placed by waiter." Don't add customer self-ordering unless explicitly requested. **Prevention:** CustomerView stays read-only for orders.

---

### Unvalidated Assumptions

1. **Assumption: The server runs Express and WebSocket on port 3001.** — Validated: `server/index.js:11` confirms `PORT = 3001`.

2. **Assumption: The existing ExpeditorEngine simulation must keep running alongside the new order flow.** — Validated: It should, as the ManagerView depends on it. But the random ticket generator (`server/index.js:92-97`) will conflict with real orders. **Fix:** Disable random ticket generation when the new order API is active, or make it togglable.

3. **Assumption: WebSocket broadcast reaches the correct customer.** — NOT validated. Current WebSocket handler (`server/index.js:21-51`) broadcasts to ALL clients. There's no per-table-code or per-session filtering. The `CustomerView.tsx:43` sends a `register` message, but nothing on the server handles it.

### Missing Acceptance Criteria

1. **AC-1:** A user can select "Customer" role, enter an optional name and table number, and enter the app without any credentials. Pass: Session is created via `/api/session` and returned. Fail: Error or no session.

2. **AC-2:** A waiter can browse the menu by cuisine, select a chef (or auto-select), add items to cart, enter a table code, and place an order. Pass: Order appears in customer's view within 2 seconds. Fail: Order never appears or errors.

3. **AC-3:** A customer sees real-time order status progression (`placed → preparing → ready → served`). Pass: Status changes within 5 seconds of server-side change. Fail: Customer must refresh to see updates.

4. **AC-4:** A customer sees estimated time remaining in minutes, and it counts down. Pass: ETA decreases roughly every minute. Fail: ETA is static or missing.

5. **AC-5:** After order is ready/served, the customer can rate: Chef Skill, Hospitality, Service Speed, Overall Experience (each 1-5 stars) + optional comment. Pass: Review is saved, chef rating is updated. Fail: Review endpoint errors or rating doesn't change.

6. **AC-6:** Chefs are displayed sorted by rating (highest first) in each cuisine category in WaiterView. Pass: After a 1-star review, chef rating decreases and sort order changes. Fail: Rating is stale.

7. **AC-7:** Manager view continues to work with real-time station/ticket/action data. Pass: All existing ManagerView metrics update live. Fail: ManagerView breaks.

8. **AC-8:** App.tsx routes users to the correct view based on their role, and allows logout to return to RoleSelect. Pass: Customer→CustomerView, Waiter→WaiterView, Manager→ManagerView, no role→RoleSelect. Fail: Wrong view rendered.

### Edge Cases

1. **Customer opens app before waiter places order** — CustomerView should show "Waiting for order" state with table code display. ✅ Already handled in `CustomerView.tsx:112-120`.

2. **Waiter enters non-existent table code** — Order is created but no customer is listening. Should still work; customer sees it when they log in with that code later.

3. **WebSocket disconnects mid-order** — Customer loses real-time updates. On reconnect, should fetch current order state via REST. ✅ `CustomerView.tsx:58-71` does an initial REST fetch on mount.

4. **Multiple orders for same table** — Should all be visible. ✅ `CustomerView.tsx:48-52` accumulates orders.

5. **Review submitted twice for same order** — Frontend tracks submitted reviews in a `Set` (`CustomerView.tsx:38`), but this is client-side only. Server has no deduplication. **Fix needed:** Server should reject duplicate reviews per orderId.

## Part 2 — Architect Plan: Implementation Roadmap

### Summary

The entire feature set (Customer role, chef selection, order tracking, reviews) is **90% built** across disconnected modules. The core work is **wiring**: connecting server modules to Express routes, connecting App.tsx to role-based pages, and connecting WebSocket broadcasts to specific sessions.

### Root Cause

The app was built in two layers that were never connected:
- **Layer 1:** The original ExpeditorEngine + Dashboard (`App.tsx`, `server/index.js`, `components/*`)
- **Layer 2:** The role-based system (`pages/*`, `context/*`, `server/chefs|menu|orders|reviews|sessions.js`)

Layer 2 is complete but unreachable.

---

### Phase 1: Server API Wiring (Priority: 🔴 CRITICAL)

**File:** `server/index.js` — Currently 125 lines, needs expansion to ~200 lines.

**What to do:**

```
1. Import all modules at top:
   - ChefManager from './chefs.js'
   - MenuManager from './menu.js'  
   - OrderManager from './orders.js'
   - ReviewManager from './reviews.js'
   - SessionManager from './sessions.js'

2. Initialize managers (after ExpeditorEngine):
   - const chefManager = new ChefManager()
   - const menuManager = new MenuManager()
   - const sessionManager = new SessionManager()
   - const orderManager = new OrderManager(expeditor, chefManager)
   - const reviewManager = new ReviewManager(chefManager)

3. Add express.json() middleware

4. Mount REST API routes:
   POST /api/session          → sessionManager.createSession(name, role, tableNumber)
   GET  /api/menu             → { items: menuManager.getMenuGrouped(), cuisineLabels: chefManager.getCuisineLabels() }
   GET  /api/chefs            → { chefs: chefManager.getChefsGrouped(), cuisineLabels: chefManager.getCuisineLabels() }
   POST /api/orders           → orderManager.createOrder({...}) + broadcast to table
   GET  /api/orders/table/:tc → orderManager.getOrdersByTableCode(tc) with ETA
   POST /api/reviews          → reviewManager.createReview({...})
   GET  /api/reviews/chef/:id → reviewManager.getReviewsByChef(id)
   GET  /api/reviews/stats    → reviewManager.getAverageRatings()

5. In the 1s simulation loop, add:
   orderManager.syncWithExpeditor()  → syncs order status with ticket status
   
6. Broadcast order updates to connected customers by table code
```

**References:**
- `server/index.js:1-12` — Current imports (needs expansion)
- `server/index.js:7` — `app` already exists (Express)
- `server/index.js:65-89` — Simulation loop (add sync call)
- `server/orders.js:159-175` — `syncWithExpeditor()` already written

### Phase 2: WebSocket Session Tracking (Priority: 🟡 HIGH)

**File:** `server/index.js` — WebSocket handler section

**What to do:**

```
1. Store connected clients with metadata:
   Map<WebSocket, { sessionId, role, tableCode }>

2. Handle 'register' message from clients:
   When a client sends { type: 'register', sessionId, role, tableCode },
   store the metadata for that WebSocket connection.

3. Add targeted broadcast function:
   broadcastToTable(tableCode, type, payload) — only sends to 
   clients registered with that table code.

4. When order status changes, call broadcastToTable() to notify
   the specific customer.
```

**References:**
- `server/index.js:18` — `clients = new Set()` → change to `Map`
- `server/index.js:21-51` — WebSocket connection handler (add register handling)
- `src/pages/CustomerView.tsx:43` — Client sends `register` message (already done)

### Trade-offs

| Decision | Pros | Cons |
|----------|------|------|
| **Keep ExpeditorEngine running alongside new order flow** | ManagerView continues working; demonstrate AI capability | Random tickets mix with real orders; metrics may be confusing |
| **No ChefView in Phase 1** | Scope contained; ship faster | Chefs can't see their orders or mark them ready (relies on auto-sync from expeditor ticket status) |
| **Client-side routing (no React Router)** | No new dependency; simple role-based switch | No URL-based navigation; refresh loses context (mitigated by localStorage session) |
| **WebSocket per-table filtering** | Customers only get their updates | More complex server logic; need to handle reconnection |
